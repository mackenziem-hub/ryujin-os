Why this exists

Memory used to drift across machines

Every machine kept its own context, and nothing reconciled them.

Session notes rode OneDrive, which forks to a per-machine copy on any concurrent edit. The durable memory in ~/.claude synced nowhere at all. Three machines, three different brains.

03 / 11
The fix

One source of truth, local files are a cache

Supabase holds the canonical brain in two tables. Every local file is rebuilt from it.

Two tables in, three rebuildable files out. Delete a local file and the next LOAD restores it. Append-only rows mean two machines saving at once become two rows, never a fork.
04 / 11
How it spreads

Every machine reads and writes the same brain

Every save is one row tagged with the machine hostname, so concurrent saves never collide. Cat's machine became the fourth spoke today and pulled the whole brain on its first LOAD.
05 / 11
The two verbs

SAVE writes one row, LOAD rebuilds

Two machines saving in the same minute become two rows, never an overwrite. That is what makes the OneDrive fork structurally impossible.

How it loads

The principle files and MEMORY.md auto-load every session. SESSION_CONTEXT.md is read on demand at LOAD. Everything is rebuilt from the two Supabase tables, so a fresh machine reaches parity in one pull.

What is next
