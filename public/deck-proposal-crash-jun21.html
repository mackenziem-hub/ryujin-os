The break was entirely in the URL routing layer. Nothing that stores or renders a proposal was affected, which is exactly why the snapshot read green the whole time.
How a proposal link works

One line in vercel.json turns /p/anything into its proposal page.

Every customer link is a short slug like /p/199-ch-frigault-e320e9. A single rewrite rule hands that slug to the renderer. Delete or misorder that rule and the catch-all underneath swallows the request into a folder that does not exist.

RULE PRESENT
/p/:slug → /proposal-v2.html?instance=:slug 200
RULE GONE
/p/:slug falls to /(.*) → /public/p/:slug 404
The data was always one fetch away. The link just had nowhere to land.
Root cause

A 9:19 PM worktree deploy overwrote the good afternoon build.

1pm to 4pm

Good deploys, Jun 20. CRM 360, the 5 proposals, the metal links. The /p/ links were verified live.

9:19 PM

Deploy jyfqyajs6 promoted to prod. No git commit attached (a CLI push from a stray worktree). Its vercel.json was missing the /p/ rule. The alias flipped to it.

overnight

Every proposal link silently 404s. No alarm, no log, no page. The product looked fine to every dashboard.

6:45 AM

Correct build redeployed (pw7ya19eo). All links back to 200.

Vercel prod promotion is last-writer-wins. The newest deploy takes the alias even when it carries older code. jyfqyajs6 was newer on the clock but stale in content.
How we pinned it

The fingerprint: the redirect still fired, the rewrite did not.

Redirect (652 to canonical)
307 fired
Rewrite (/p/ to page)

Vercel evaluates redirects before rewrites. A redirect in the same config still 307'd, which proved the file was live and reachable. The /p/ rewrite next to it returned 404, which proved that one rule was gone. That single split ruled out DNS, the database, the renderer, and an expired link. It could only be the routing rule.

Why nobody caught it for hours

It ran dark for hours because a customer-facing 404 alarms nothing.

01

No error telemetry on the proposal path. There is zero Sentry or equivalent watching customer pages. A 404 served to a prospect is invisible on our side.

02

The snapshot read green. Revenue, pipeline, and proposal data were all healthy, because the data was never the problem. The health checks watch the wrong layer.

03

A human click was the entire detection system. You opening Diane's link this morning is what surfaced it. Detection took roughly nine hours. The fix took fifteen minutes.

The fix

Redeploying the correct main build restored all links in 15 minutes.

There was no code change to make. The main branch already held the correct vercel.json with the /p/ rule. The fix was to deploy it cleanly and re-promote the alias, then verify each link returned 200.

Stopping the next one

Two guardrails stop the next clobber.

01

Deploy prod from one place only. A clean worktree off origin/main, never from a stray worktree. The clobber happened because many parallel builds can each promote the alias.

02

Smoke-test after every prod deploy. Curl one /p/ link plus the key routes and fail loud if anything is not 200. This outage ends in two minutes, not nine hours, with that check.

03

Stand up Sentry on the proposal and API paths. This is the second outage it would have caught and the standing infra gap from the last two sessions.

D E C I D E
Wire a post-deploy smoke check (curl /p/ plus key routes, SMS on fail), and finally stand up Sentry?
Recommendation: Yes to the smoke check today, it is cheap and I can wire it into a Windows task this morning. Sentry is the bigger lift but it is the recurring gap. The single-deployer discipline costs nothing and prevents the cause directly.

02

Parallel worktree deploys are the real risk. Many concurrent builds, last-writer-wins prod, and no smoke test add up to a silent outage waiting to happen. This is the second clobber on record.

03

04

One discipline beats ten fixes. Deploy from one clean place, smoke-test every time. Cheaper than any monitoring stack and it kills the cause.
