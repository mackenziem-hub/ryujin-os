## TOP 10 (fix this week, in order)

### 1. CRITICAL — `proposal-accept.js` accepts `estimateId` without share token
**File:** `api/proposal-accept.js:163-165`
The author wrote the comment "do not trust estimateId from client alone" then immediately implemented exactly that fallback: `shareToken ? .eq('share_token', shareToken) : .eq('id', estimateId)`. Any client passing only `estimateId` can accept any tenant's quote. Public endpoint.
**Fix:** Make `shareToken` mandatory. Remove the `.eq('id', estimateId)` branch entirely. If a legacy caller needs estimateId-only path, gate it behind `requireTenant`.
**Effort:** 5 min.

### 2. CRITICAL — `api/chat.js` system prompt is 50-100KB, sent uncached on every message
**File:** `api/chat.js:2243-2483`
BASE_PROMPT (~12KB) + 6 agent personas (~18KB) + snapshot (variable, often 10-50KB) + memory + preferences, all sent every request to Sonnet-4-6 with 3000-token thinking budget enabled by default. Zero use of Anthropic's prompt caching. Likely costing several hundred dollars/month in waste on the *system prompt alone*.
**Fix:** Mark BASE_PROMPT + active persona as `cache_control: { type: 'ephemeral' }`. Only include the active agent persona, not all 6. Disable thinking budget except for explicit "think this through" requests. See `claude-api` skill for the exact caching pattern.
**Effort:** 60-90 min, but pays back in days.

### 3. CRITICAL — `admin.html` Shenron chat call missing `x-tenant-id` header
**File:** `public/admin.html:4522-4524`
`fetch('/api/chat', { headers: { 'Content-Type': 'application/json' } })` — no tenant header. `requireTenant` will reject (or default to whatever fallback resolves first). Today this works because there's only one tenant, but the chat is silently broken for any white-label deployment.
**Fix:** Add `'x-tenant-id': TENANT` to the headers (TENANT is already declared at line 680).
**Effort:** 1 line.

### 4. HIGH — `marketing-render.js` race condition lets the same clip render twice
**File:** `api/marketing-render.js:59` + `lib/marketingRenderer.js:29`
Classic TOCTOU: read `clip.status`, then call `renderClip()` which sets status='rendering'. Two concurrent invocations (e.g., upload kickoff + cron `?next=1` firing at the same moment) both pass the check and both render. Wastes ffmpeg time, Whisper credits, Vercel function-seconds, Blob storage, and Claude tokens.
**Fix:** Atomic update: `update set status='rendering' where id=X and status='queued' returning *`. If 0 rows returned, somebody else got it. Refuse cleanly.
**Effort:** 15 min.

### 5. HIGH — 15+ production pages hardcode `'plus-ultra'` as tenant
**Files:** `production-jobs.html:133`, `production-materials.html:254-255`, `production-workorders.html:195`, `production-paysheet.html:189`, `job.html:140+`, `sub-portal.html`, `sales-proposal.html`, `marketing-schedule.html:171`, others.
Hard wall against any second tenant. White-label SaaS blocker.
**Fix:** Single pattern: `const TENANT = window.RyujinTenant?.get?.()?.slug || 'plus-ultra'`. Already used in `marketing-creatives.html:166` — copy that pattern everywhere. Could be one search/replace pass + smoke test.
**Effort:** 30-45 min, mostly mechanical.

### 6. HIGH — `api/assistant.js` uses Sonnet-4 for a 300-token chat reply
**File:** `api/assistant.js:114`
Model is `claude-sonnet-4-20250514`, max_tokens 300, system prompt ~380 bytes, task is FAQ + navigation. Sonnet-4 is ~15× the per-token cost of Haiku-4-5 for a job that doesn't need it. Every visitor chat on the proposal page pays this premium.
**Fix:** Switch to `claude-haiku-4-5-20251001`. Spot-check 5 common questions before/after. Memory note about latest models is in your auto-memory; check there for the right ID.
**Effort:** 5 min + test.

### 7. HIGH — `marketing-reconcile.js` (built tonight) — N+1 GHL calls, 60s timeout
**File:** `api/marketing-reconcile.js` + `vercel.json:33-35`
At default `limit=50` and ~500ms per GHL call, runs ~25s — fits. At `limit=200` (HARD_LIMIT) or slow GHL, exceeds the 60s function timeout and the cron silently dies mid-batch. **Self-audit catch.**
**Fix:** Wrap the per-row loop in `Promise.all` chunks of 8-10. Bonus: track per-chunk failures so one slow GHL response doesn't block the rest.
**Effort:** 20 min.

### 8. HIGH — `api/sub-auth.js` job lookup is correctly tenant-scoped (FALSE POSITIVE)
Backend agent flagged this as critical multi-tenant leak. Re-read line 44: `.eq('tenant_id', tenantId).eq('subcontractor_id', sub_id)` — it IS scoped. **Skip this one.**

### 9. HIGH — Stale data on dashboards: no visibility-change refetch
**Files:** `proposal-history.html:366`, `production-jobs.html:177`, `marketing-schedule.html`, `admin.html` (caches estData/custData/offersCache forever)
Each page fetches once on init and never re-fetches. A new proposal sent in another tab → list stays stale until F5. UX feels broken; data feels untrusted.
**Fix:** One-line per page: `document.addEventListener('visibilitychange', () => { if (!document.hidden) refetch(); })`. Could even centralize as a `RyujinRefresh.on(refetchFn)` utility.
**Effort:** 30 min for all four.

### 10. MEDIUM — Silent fallbacks in form parsing mask invalid input
**Example:** `admin-tenant.html:263-264` — `parseFloat($('f_subRate').value) || 0`. Empty or non-numeric input silently saves as 0, then HST/sub rates are wrong, then quotes are wrong. This is the same *category* of bug that produced today's 95 Cornhill 1-roll-of-I&W incident — silent fallbacks in calculation paths.
**Fix:** Audit every `parseFloat(x) || 0` and `Number(x) || 0` in form handlers. Reject the save and surface a specific error if a required numeric field doesn't parse.
**Effort:** 30-60 min depending on how many.
